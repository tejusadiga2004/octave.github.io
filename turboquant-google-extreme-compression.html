TurboQuant: Google's Answer to the AI Memory Crisis

Google Research's TurboQuant introduces a theoretically grounded, zero-overhead vector quantization algorithm that compresses LLM key-value caches to as low as 3 bits with no accuracy loss and up to 8x speedup on H100 GPUs — a genuine breakthrough for memory-constrained AI deployment.

Tejus Adiga M Thu 02 April 2026
machine learning quantization llm kv cache memory compression google vector search transformers

Google Research's TurboQuant, presented at ICLR 2026, is a compression algorithm that combines two elegant mathematical ideas — PolarQuant and the Quantized Johnson-Lindenstrauss transform — to slash the memory cost of large language model inference with zero accuracy loss and near-zero implementation overhead. This post breaks down the problem it solves, how it works under the hood, and what it means for the future of AI at scale.

1. The Memory Explosion: How LLMs Became Memory Hungry

Modern large language models (LLMs) like Gemini, GPT-4, and Llama-3 operate on the Transformer architecture, in which the core computation is self-attention. To produce each output token, the model must compare the current query against every prior key and value in the input sequence. As context windows grow from thousands to millions of tokens, this creates a formidable problem: you have to store all of those keys and values somewhere.

That somewhere is the Key-Value (KV) cache — a data structure that holds the intermediate key and value vectors computed from every prior token in the active context. The KV cache is critical for efficient autoregressive generation; without it, the model would recompute the entire sequence from scratch for every new token it generates.

The memory impact is significant. For a model with \(L\) layers, \(H\) attention heads, head dimension \(d\), and a context of \(N\) tokens, the KV cache footprint scales as:

\[\text{KV memory} = 2 \times L \times H \times d \times N \times \text{bytes_per_element}\]

In practice, with 32-bit floats, this turns gigabytes for a moderate context window into tens of gigabytes for a long one. Consider a model with 32 layers, 32 heads, and head dimension 128 running a 128K token context:

\[2 \times 32 \times 32 \times 128 \times 131072 \times 4 \approx 137 \text{ GB}\]

That is far beyond what fits on a single GPU. Even with the shift to 16-bit floats, the picture only halves. And as context windows push toward millions of tokens — necessary for tasks like document understanding, long-horizon reasoning, and genomics — the problem grows proportionally. The KV cache has become the primary memory bottleneck in modern LLM inference.

Beyond LLMs, the same explosion appears in vector search engines. Products like Google Search increasingly rely on retrieval over billions of high-dimensional vector embeddings. Storing and searching those vectors in full 32-bit precision is simply untenable at scale.

2. Prior Attempts to Solve the Memory Problem

The AI community has pursued several strategies to contain the KV cache and vector storage problem. None have been fully satisfying.

Scalar Quantization

The most direct approach is to reduce the numerical precision of stored values. Storing keys and values in 8-bit integers (INT8) or 4-bit integers (INT4) cuts memory by 4–8x over FP32. This is straightforward to implement and hardware-friendly, relying on the same integer arithmetic units found in modern GPUs.

The catch is quantization overhead. To correctly decode a quantized value, the model needs to store the scale factors and zero-points used during encoding. Doing this per-tensor is coarse and damages accuracy. Doing it per-channel or per-block is accurate but bloated — those quantization constants can add 1–2 extra effective bits per stored value, eating a significant fraction of the savings.

KIVI

KIVI (Key-Value cache Inference via In-situ Quantization) is a practical KV cache compression baseline that applies per-channel quantization to keys and values, keeping the most recent tokens in full precision and quantizing older context. It achieves solid compression ratios with reasonable accuracy but still relies on stored quantization constants and requires careful tuning of the residual window.

Product Quantization (PQ)

Product Quantization is a classical vector compression technique popular in approximate nearest-neighbour (ANN) search. It splits each high-dimensional vector into sub-vectors, learns a small codebook for each sub-dimension, and encodes each sub-vector as an index into its codebook. During retrieval, distances are approximated using precomputed lookup tables.

PQ achieves strong compression but has important limitations. Codebooks must be learned offline from training data, making the approach data-dependent. This means it cannot be applied in an online setting (e.g., dynamically growing KV caches during decoding) without incurring expensive re-indexing. Codebook storage also adds memory overhead.

RaBitQ and Related Methods

RaBitQ and similar binary vector quantization schemes push compression to 1 bit per dimension using random orthogonal projections before quantization. These methods achieve near-zero overhead and are data-oblivious, but a single bit per dimension causes significant distortion at low dimensions. Extensions like RabbiQ improve accuracy through multi-bit representations but introduce data-specific codebooks or hyperparameter tuning.

The Common Failure Mode

All of the above methods share a structural weakness: they trade off one of three desirable properties — no memory overhead, high accuracy, and data-oblivious (online) applicability — against the others. TurboQuant's design goal was to achieve all three simultaneously.

3. Introducing TurboQuant

TurboQuant was introduced by Amir Zandieh (Research Scientist) and Vahab Mirrokni (VP and Google Fellow) at Google Research, with collaborators from Google DeepMind, KAIST, and NYU. It was published in March 2026 and accepted to ICLR 2026.

At its core, TurboQuant is a vector quantization algorithm for compressing high-dimensional vectors — specifically the keys and values in transformer attention layers and the indexed vectors in large-scale ANN search engines. Its defining characteristics are:

  • Zero quantization overhead: no scale factors, zero-points, or codebooks need to be stored.
  • Data-oblivious: works online without needing to see the dataset in advance.
  • Theoretically optimal: provably achieves near-optimal distortion rates, operating near theoretical lower bounds.
  • No fine-tuning required: plug-in compression at inference time, no retraining.
  • Significant precision reduction: quantizes KV caches to as low as 3 bits per value.

TurboQuant achieves this by synthesising two component algorithms — QJL (Quantized Johnson-Lindenstrauss) and PolarQuant — into a unified two-stage pipeline. Understanding TurboQuant fully requires understanding both components.

4. Deep Dive into the TurboQuant Architecture

4.1 The Core Problem: Overhead-Free Quantization

Every classical quantization scheme faces a fundamental tension. To accurately reconstruct a vector from its quantized form, you need reference information: the minimum and maximum of the values in the block, a scale factor, a learned code, etc. These are the quantization constants, and they must be stored alongside the compressed data.

If you quantize a 128-dimensional vector to 4 bits per element with per-block constants, you spend 512 bits on actual data and perhaps 32–64 bits on constants — an overhead of 6–12%. At scale, over billions of vectors, this adds up. More critically, per-block constants must be computed and fetched at decoding time, adding latency.

TurboQuant's insight is to transform the data before quantization in such a way that the transformed representation no longer requires external reference information to be meaningful and reconstructable. Both QJL and PolarQuant achieve this through principled geometric transformations.

4.2 QJL: The Quantized Johnson-Lindenstrauss Transform

The Johnson-Lindenstrauss (JL) lemma is a classical result in high-dimensional geometry. It states that any set of \(n\) points in \(\mathbb{R}^d\) can be embedded into \(\mathbb{R}^k\) (where \(k = O(\log n / \varepsilon^2)\)) using a random linear map, while approximately preserving all pairwise distances to within factor \((1 \pm \varepsilon)\).

A particularly efficient implementation uses a random matrix \(S \in \mathbb{R}^{k \times d}\) with i.i.d. \(\pm 1 / \sqrt{k}\) entries (or structured fast variants using Hadamard transforms). Applying \(S\) to a vector \(v\) gives \(Sv\), a lower-dimensional sketch that preserves inner products:

\[\langle u, v \rangle \approx \langle Su, Sv \rangle\]

QJL takes this one step further: after computing the JL sketch, it discards all magnitude information and retains only the sign bit of each component:

\[\text{QJL}(v) = \text{sign}(Sv) \in \{-1, +1\}^k\]

This gives exactly 1 bit per output dimension with zero overhead — no scale factors, no codebooks, nothing. The key insight is that the sign of a random projection encodes probabilistic distance information. For two unit vectors \(u\) and \(v\), the probability that a random hyperplane separates them equals \(\frac{\theta}{\pi}\), where \(\theta = \arccos(\langle u, v \rangle)\). This gives an unbiased estimator of inner products from sign bits alone.

To compute attention scores, QJL uses an asymmetric estimator: queries are kept at full precision while keys are stored in sign-bit form. The attention score between a full-precision query \(q\) and a sign-quantized key \(\hat{k} = \text{sign}(Sk)\) is estimated as:

\[\langle q, k \rangle \approx \frac{\sqrt{\pi}}{2} \frac{1}{m} \sum_{i=1}^{m} q^T S^T_i \hat{k}_i\]

This estimator is unbiased, has bounded variance, and introduces zero additional memory allocation — the only thing stored for each key is a bit vector.

4.3 PolarQuant: Compression via Polar Decomposition

While QJL achieves zero overhead through extreme 1-bit quantization, it has high distortion when used alone at low bit counts. PolarQuant takes a complementary approach: it achieves zero overhead through a change of coordinate system rather than extreme precision reduction.

Classical Cartesian quantisation of a vector \(v = (v_1, v_2, \ldots, v_d)\) requires knowing the range of each coordinate to set the quantization scale. These ranges vary by vector and must therefore be stored — classic overhead.

PolarQuant converts the vector from Cartesian to polar coordinates in a recursive fashion:

  1. Pair coordinates: Group the \(d\) components into \(d/2\) pairs \((v_1, v_2), (v_3, v_4), \ldots\)
  2. Compute polar form: For each pair \((x, y)\), compute the radius \(r = \sqrt{x^2 + y^2}\) and angle \(\theta = \arctan(y/x)\).
  3. Recurse on radii: Collect all the radii \(r_1, r_2, \ldots, r_{d/2}\) and repeat the pairing-and-polarisation step.
  4. Terminate: Continue recursively until a single final radius remains. This single scalar captures the overall magnitude (norm) of the original vector.

The result is one final radius and \(d - 1\) angles.

The critical observation is that a randomly rotated high-dimensional vector has angles that follow a known, highly concentrated distribution — specifically, they are approximately uniformly distributed over the hypersphere's surface. This means:

  • The quantizer for the angles does not need to adapt to the data; a fixed grid suffices.
  • No per-vector normalisation constants need to be stored alongside the angles.
  • The only stored extra value is the final scalar radius, which requires just a few bits and can itself be efficiently quantised.

This eliminates the overhead that plagues Cartesian quantisation while delivering much higher fidelity than QJL at the same bit count.

4.4 Putting It Together: The TurboQuant Pipeline

TurboQuant combines both components in a two-stage pipeline:

Stage 1 — High-quality bulk compression with PolarQuant: First, TurboQuant applies a random rotation to the input vector. This is the key preprocessing step: by isotropically rotating the data, the coordinates become exchangeable and the distributional assumptions of PolarQuant are met. Then PolarQuant is applied, using the bulk of the available bit budget to encode the vector's structure with high fidelity and zero overhead.

Stage 2 — Residual error correction with QJL: No compression is perfect. After PolarQuant, a small residual error vector remains — it is the difference between the true vector and its PolarQuant reconstruction. TurboQuant allocates just 1 additional bit per dimension to apply QJL to this residual. Because the error is small and zero-mean (PolarQuant is an unbiased estimator), QJL's probabilistic estimator is extremely effective at removing any remaining systematic bias from the attention score computation.

The combined scheme achieves: - The zero-overhead guarantee of QJL - The high-fidelity encoding of PolarQuant - Provably optimal distortion rates - A pipeline that is simple, fast, and requires no training data

At 4 bits per value (3 bits PolarQuant + 1 bit QJL), TurboQuant approaches the theoretical optimum for inner product preservation. At 3 bits, it remains essentially lossless for attention computation.

5. A Practical Example of TurboQuant

To make the architecture concrete, consider applying TurboQuant to the Llama-3.1-8B-Instruct model at inference time on an H100 GPU.

The Setup

Llama-3.1-8B has: - 32 layers - 32 attention heads (using GQA with 8 key-value heads) - Head dimension 128

For a long-context task with a 32K token sequence, storing the KV cache in FP16 requires approximately:

\[2 \times 32 \times 8 \times 128 \times 32768 \times 2 \approx 4.3 \text{ GB}\]

At 3-bit TurboQuant that becomes roughly 0.7 GB — a 6× reduction.

Applying TurboQuant

At each layer, as new key and value vectors are computed (each a 128-dimensional FP16 vector):

  1. Rotate: Multiply the key vector by a pre-generated random orthogonal matrix \(R \in \mathbb{R}^{128 \times 128}\). This step is cheap — the matrix can be implemented as a structured Hadamard transform in \(O(d \log d)\) rather than \(O(d^2)\).
  2. PolarQuant encode: Apply the recursive polar decomposition, allocating 2 bits to each angle. Store the compressed bit representation.
  3. QJL encode residual: Decode the PolarQuant approximation, subtract it from the original, then apply sign quantization to the residual using a pre-generated random projection matrix (shared across all vectors, so it only needs to be stored once).

During attention computation, for each query \(q\):

  1. PolarQuant decode: Reconstruct an approximate key from its stored polar bits. Compute a partial attention score.
  2. QJL correction: Apply the asymmetric QJL estimator to the stored residual bits to correct the attention score.
  3. Softmax and aggregation: Proceed with standard attention aggregation using the corrected scores.

Results on Standard Benchmarks

Google Research evaluated TurboQuant on the LongBench, Needle-in-a-Haystack, ZeroSCROLLS, RULER, and L-Eval benchmarks using Gemma and Mistral models. On Llama-3.1-8B-Instruct:

Method Bits LongBench Score KV Memory Reduction
Baseline (FP16) 16 100% 1×
KIVI 4 ~98% 4×
PolarQuant 3 ~99% ~5×
TurboQuant 3 ~100% 6×

On the Needle-in-a-Haystack task — which requires retrieving a single specific fact from a large context, an extreme test of key-value fidelity — TurboQuant achieves perfect downstream results while compressing to 3 bits. KIVI at the same bit count shows measurable degradation.

The runtime story is equally compelling. Because 3-bit and 4-bit integer arithmetic is fundamentally faster than 32-bit floating-point, the compressed attention computation is faster than the unquantized baseline. On H100 GPUs, 4-bit TurboQuant achieves up to 8× faster attention logit computation compared to 32-bit unquantized keys, measured against a highly optimised JAX baseline. Even 3-bit TurboQuant delivers substantial speedup, meaning the compression pays dividends not just in memory but in wall-clock latency.

For vector search, TurboQuant was evaluated on the GloVe dataset (200-dimensional embeddings) against PQ and RabbiQ using the 1@k recall ratio. TurboQuant achieves optimal recall while using no training data and no precomputed codebooks — a remarkable result given that PQ and RabbiQ require expensive offline preprocessing.

6. Opportunities to Optimise Further

TurboQuant as published is already near the theoretical optimum for single-vector quantization. But it opens a number of directions for further improvement.

Structured Random Rotations

TurboQuant's preprocessing step uses a random orthogonal matrix to isotropise the data. In the paper, this is implemented using a fast Walsh-Hadamard transform, which costs \(O(d \log d)\) rather than \(O(d^2)\). Future work could explore whether data-adaptive rotations (e.g., learned via a lightweight online update) could improve the match between real KV vectors and the polar distribution assumption, yielding better accuracy at ultra-low bit counts (1–2 bits).

Mixed-Precision Allocation

Not all attention heads are equally sensitive to quantization error. Some heads perform sharp, focused attention (highly sensitive to key precision) while others compute diffuse, broad attention. An adaptive bit allocation scheme that identifies head types online and assigns different bit budgets accordingly could squeeze additional efficiency from the same total memory budget, with the framework remaining end-to-end overhead-free through TurboQuant's zero-constant encoding.

Extension to Values

TurboQuant as evaluated focuses primarily on key quantization, since keys directly affect the attention score computation. But value quantization is also a significant memory consumer. Values can be compressed more aggressively (since small errors in values produce small errors in the weighted sum output, not catastrophic attention misalignment) and TurboQuant's machinery applies equally well to the value cache.

Multi-Query and Grouped-Query Attention

In modern LLMs, Grouped-Query Attention (GQA) and Multi-Query Attention (MQA) share key-value heads across multiple query heads, already reducing the KV cache footprint by a factor of the grouping ratio. TurboQuant is fully compatible with these architectures and the two optimisations compound: GQA reduces the number of KV vectors; TurboQuant reduces the bits per vector.

Vector Search at Google Scale

Beyond LLMs, TurboQuant has direct implications for Google's search infrastructure, which must perform nearest-neighbour retrieval over billions of vectors in milliseconds. Existing production systems rely on PQ and its variants, which require offline index construction and periodic re-training. TurboQuant's data-oblivious property enables streaming vector indexing — new items can be added to the index without recomputing the codebook or reindexing existing data. This is transformative for dynamic, freshness-sensitive search applications.

Cascaded Residual Quantization

The TurboQuant pipeline (bulk encoding + 1-bit residual correction) is naturally extensible to a multi-stage residual cascade: encode, correct with QJL, compute residual of corrected approximation, encode again, and so on. Each stage adds a small number of bits and progressively refines the approximation. This residual quantization approach is well-studied in the PQ literature; merging it with TurboQuant's zero-overhead guarantees is a natural next step.

Summary

TurboQuant is a genuinely foundational contribution. It is not an incremental engineering improvement but a theoretically grounded algorithm that provably solves a problem — zero-overhead, data-oblivious, high-fidelity vector quantization — that has blocked progress in both LLM inference and large-scale vector search.

For practitioners, the immediate takeaway is striking: a 3-bit KV cache with no fine-tuning, no accuracy loss, a 6× memory reduction, and up to an 8× speedup on modern GPUs. For researchers, TurboQuant's synthesis of Johnson-Lindenstrauss theory with polar coordinate representations opens a rich design space for future compression algorithms. And for Google's infrastructure, it provides a principled foundation for semantic search and LLM serving at a scale no other method can match without sacrificing accuracy.
